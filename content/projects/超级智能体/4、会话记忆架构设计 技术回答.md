# 第一层：整体架构设计

## Q1：为什么要设计会话记忆系统？

### 回答

大模型本身是无状态的，每次调用只能看到当前 Prompt。

如果直接把全部历史对话拼进去，会产生三个问题：

1. Token 成本线性增长
2. 上下文窗口最终会溢出
3. 早期历史信息会稀释当前问题的注意力

因此我设计了一个可插拔式会话记忆架构，在：

- Token 成本
- 上下文完整性
- 响应性能

之间做平衡。

系统支持：

- 无记忆模式
- 滑动窗口模式
- 增量摘要压缩模式

根据会话长度自动切换。

---

## Q2：为什么不用直接滑动窗口？

### 回答

滑动窗口只能保留最近 N 轮。

例如：

```
第1轮：项目名称是智能客服系统第50轮：请问我刚刚那个项目怎么设计的？
```

如果窗口只保留最近 4 轮：

```
第1轮的信息已经丢失
```

模型无法回答。

因此需要长期记忆。

我的方案：

```
长期摘要+最近窗口原文
```

例如：

```
【长期摘要】项目名称：智能客服系统技术栈：Spring AI + ES【最近原文】最近4轮对话
```

这样既保留长期信息，又控制 Token。

# 第二层：摘要压缩设计

---

## Q3：为什么选择增量摘要，而不是全量重摘要？

### 回答

主要是性能问题。

假设：

```
100轮对话
```

如果每次重新压缩：

```
第101轮：重新读取100轮重新调用LLM重新生成摘要
```

复杂度：

```
O(总轮次)
```

随着会话增长成本越来越高。

我的方案是：

```
已有摘要+新增批次→ 合并生成新摘要
```

例如：

```
摘要覆盖：1~60轮新增：61~66轮
```

只处理：

```
摘要 + 61~66轮
```

复杂度：

```
O(新增轮次)
```

长期运行成本更低。

---

## Q4：为什么摘要设计成 JSON 而不是纯文本？

### 回答

因为后续步骤需要结构化消费。

如果是纯文本：

```
用户目标...系统模块...待解决问题...
```

后续规划器只能再让 LLM 解析一次。

增加：

- Token
- 延迟
- 幻觉风险

因此我设计了：

```
{  "conversation_goal": "",  "stable_facts": [],  "resolved_points": [],  "pending_questions": [],  "retrieval_hints": []}
```

后续：

Step2 改写器

Step3 路由器

Step4 Planner

可以直接读取对应字段。

例如：

```
payload.getPendingQuestions()
```

无需再次解析。

# 第三层：Token优化

---

## Q5：如何控制 Token 成本？

### 回答

主要做了三层控制。

### 第一层：滑动窗口

只保留最近：

```
keepRecentTurns=4
```

轮原文。

---

### 第二层：摘要压缩

旧轮次压缩为：

```
summary
```

控制在：

```
summaryMaxChars=1400
```

以内。

---

### 第三层：预算分配

规划上下文预算：

```
planningHistoryMaxChars=1600
```

按比例切分：

```
最近原文 65%摘要 35%
```

即：

```
1040字符+560字符
```

避免某一部分独占全部 Token。

---

## Q6：为什么用字符数而不是Tokenizer精确计算？

### 回答

因为需要兼顾性能。

Tokenizer计算：

```
tokenizer.encode()
```

每次都需要执行编码。

在高并发场景下开销较大。

中文场景经验值：

```
1 Token ≈ 4个中文字符
```

因此采用：

```
estimatedTokens =ceil(content.length()/4.0)
```

进行保守估算。

误差可接受。

但性能更高。

# 第四层：并发安全

---

## Q7：多个请求同时触发摘要压缩怎么办？

### 回答

存在并发覆盖风险。

例如：

```
线程A线程B
```

同时压缩同一会话。

可能出现：

```
A写入摘要B覆盖A结果
```

因此我做了两层保护。

---

### 第一层

会话级互斥

```
ConcurrentHashMap.newKeySet()
```

维护：

```
refreshingConversationIds
```

同一个会话只允许一个线程压缩。

---

### 第二层

乐观锁校验

写入前重新查询：

```
coveredExchangeId
```

如果数据库已经存在更大的版本：

```
直接放弃写入
```

防止旧摘要覆盖新摘要。

---

## Q8：为什么不用分布式锁？

### 回答

目前摘要刷新发生在单服务节点内部。

使用：

```
ConcurrentHashMap
```

成本最低。

如果后续扩展为多实例部署：

可以升级为：

```
Redis Redisson Lock
```

或者：

```
version字段CAS更新
```

实现跨节点同步。

# 第五层：异常处理

---

## Q9：LLM摘要失败怎么办？

### 回答

不能因为摘要失败影响整个会话。

因此设计了：

```
LLM+Fallback
```

双链路。

流程：

```
LLM压缩      ↓失败      ↓fallbackMerge
```

---

fallback会：

### summary

保留旧摘要

拼接新增关键信息

---

### pending_questions

继续累计

---

### retrieval_hints

正则提取关键词

例如：

```
SpringBootRedisMySQL
```

保证系统可用性。

---

## Q10：为什么摘要刷新放到异步？

### 回答

因为摘要压缩涉及一次额外 LLM 调用。

如果同步执行：

```
用户等待↓回答生成↓摘要压缩↓返回
```

RT会明显增加。

因此改成：

```
返回用户↓异步刷新摘要
```

用户无感知。

# 第六层：三路历史分发（高频面试题）

---

## Q11：为什么要拆成三路历史，而不是统一传给所有模块？

### 回答

因为不同阶段对历史信息需求不同。

### Step2 问题改写

需要：

```
长期背景+最近原文
```

---

### Step3 路由决策

需要：

```
用户目标业务事实待解决问题
```

更适合结构化信息。

---

### Step5 回答生成

需要：

```
最近用户问题
```

而不是大量旧摘要。

否则容易产生：

```
上下文污染
```

所以拆成：

```
historySummaryhistoryPlanningContextanswerHistoryContext
```

三条链路。

---

## Q12：什么叫上下文污染？

### 回答

例如用户：

```
前面讨论Java面试
```

后来切换到：

```
Redis缓存一致性
```

如果把全部历史都送进去：

模型可能持续关注：

```
JVMGC类加载
```

导致回答偏离当前问题。

因此：

回答阶段只保留：

```
answerRecentTranscript
```

即最近用户问题。

减少干扰。

# 第七层：项目亮点题（面试官最喜欢）

---

## Q13：你觉得这个记忆系统最大的亮点是什么？

### 回答

我认为有四个亮点：

### 1

增量摘要

避免全量重压缩。

复杂度从：

```
O(总轮次)
```

降低到：

```
O(新增轮次)
```

---

### 2

结构化摘要

不是文本。

而是：

```
goalfactspendinghints
```

方便下游直接消费。

---

### 3

三路历史分发

不同模块使用不同上下文。

避免上下文污染。

---

### 4

异步压缩 + 乐观锁

保证：

- 用户无感知
- 不阻塞请求
- 不出现摘要覆盖

---

# 压轴题（P8级别追问）

## Q14：如果会话达到 10000 轮怎么办？

这是非常容易问到的架构题。

### 回答

我会采用分层记忆架构：

```
L1：最近窗口L2：长期摘要L3：向量记忆
```

结构：

```
最近4轮原文+摘要+向量检索历史记忆
```

对于极早期的重要信息：

例如：

```
用户项目名称系统架构业务规则
```

存入向量库。

当前问题先检索相关记忆：

```
Recall Memory
```

再与摘要一起送入模型。

这样即使：

```
10000轮100000轮
```

也不会因为上下文窗口限制而丢失重要历史。
