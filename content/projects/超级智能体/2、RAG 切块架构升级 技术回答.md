## Q1：为什么要做父子块架构？直接切块不行吗？

### 回答

传统RAG通常只有一层Chunk。

例如：

```
第一章Spring启动流程......BeanFactory初始化......AOP代理创建
```

切成：

```
Chunk1Chunk2Chunk3
```

问题在于：

### Chunk太小

检索精准：

```
AOP代理创建
```

能命中。

但送给LLM时：

```
只有AOP代理创建这一段
```

缺少上下文。

模型不知道：

```
这是Spring启动流程中的哪个阶段？和BeanFactory有什么关系？
```

---

### Chunk太大

例如：

```
整个章节3000字
```

语义完整。

但向量检索效果下降。

因为：

```
Embedding被大量无关内容稀释
```

召回率降低。

---

所以设计：

```
Parent Block (~2200字)      ↓Child Chunk (~800字)
```

职责分离：

```
Child:负责检索Parent:负责补全上下文
```

即：

```
检索精度+上下文完整性
```

同时获得。

---

# Q2：为什么选择800和2200？

### 回答

本质是Embedding模型与LLM上下文窗口之间的平衡。

经过测试：

### 子块

小于500：

```
上下文不足
```

大于1000：

```
Embedding区分度下降
```

因此：

```
700~900
```

最稳定。

最终默认：

```
800
```

---

### 父块

需要覆盖：

```
完整章节完整步骤完整知识点
```

同时避免：

```
Prompt爆炸
```

因此：

```
2000~2500
```

比较合理。

最终：

```
2200
```

## Q3：为什么需要四种切块策略？

### 回答

因为不同文档质量差异极大。

例如：

### Markdown

```
# 第一章## Spring
```

适合：

```
STRUCTURE
```

---

### 小说

```
纯文本
```

适合：

```
RECURSIVE
```

---

### 技术文档

```
有段落有语义边界
```

适合：

```
SEMANTIC
```

---

### OCR扫描件

```
格式全乱
```

适合：

```
LLM
```

---

所以采用：

```
Strategy Chain
```

动态组合。

而不是：

```
一个切块算法打天下
```

# Q4：为什么语义切分选择Jaccard？

### 回答

主要考虑：

```
速度成本
```

---

如果使用Embedding：

```
每个句子生成向量
```

成本巨大。

例如：

```
1000句
```

需要：

```
999次向量比较
```

---

Jaccard：

```
Set交集/Set并集
```

复杂度低。

无需模型调用。

---

实现：

```
similarity =intersection.size()/union.size();
```

---

当：

```
similarity < 0.18
```

说明主题发生变化。

触发切块。

---

# Q5：为什么LLM切块还要递归切块兜底？

### 回答

因为LLM不可靠。

可能出现：

### 情况1

返回：

```
{ xxx}
```

格式错误。

---

### 情况2

超长文本。

模型截断。

---

### 情况3

网络失败。

---

所以：

```
LLM ↓失败 ↓Semantic ↓失败 ↓Recursive
```

形成降级链路。

保证：

```
100%能完成切块
```

而不是：

```
因为AI失败导致整个文档无法入库
```

## Q6：为什么设计PRIMARY、OPTIMIZE、FALLBACK？

### 回答

为了让策略具备可编排能力。

例如：

```
PRIMARYSTRUCTURE
```

先按章节切。

得到：

```
Spring启动流程
```

---

再：

```
OPTIMIZESEMANTIC
```

进一步优化。

---

最后：

```
FALLBACKRECURSIVE
```

处理异常大块。

---

形成：

```
STRUCTURE →SEMANTIC →RECURSIVE
```

流水线。

---

如果以后新增：

```
GraphChunkStrategy
```

只需要：

```
applyStrategy()
```

注册即可。

无需修改主流程。

符合：

```
开闭原则
```

## Q7：为什么只向量检索不够？

### 回答

Embedding存在天然缺陷。

例如：

用户问：

```
Java 21 虚拟线程
```

文档中：

```
Virtual Thread
```

可以召回。

---

但如果问：

```
ErrorCode=10086
```

Embedding效果很差。

---

关键词检索：

```
ES
```

非常擅长：

```
编号类名接口名配置项
```

---

因此采用：

```
Vector+Keyword
```

双通道。

---

## Q8：为什么用RRF而不是简单加权？

### 回答

因为：

```
向量分数
```

和

```
ES BM25分数
```

量纲不同。

不能直接加。

---

RRF公式：

RRF=∑1K+rRRF = \sum \frac1{K+r}RRF=∑K+r1​

这里：

```
K=60
```

---

只关心：

```
排名
```

不关心：

```
具体分数
```

非常适合多检索器融合。

### 回答

因为它解决了：

```

```

---

通过：

```
parent_block_id
```

找到：

```

```

整个父块。

---

构造：

```
完整父块命中子片段:child#15 ...
```

---

这样模型同时获得：

```
完整上下文+精准命中点
```

---

这是整个架构最大的创新点。

## Q10：为什么父块分数要重新计算？

### 回答

因为：

```
多个子块命中同一父块
```

是强相关信号。

例如：

```
父块A命中1个子块父块B命中4个子块
```

显然：

```
父块B更相关
```

---

因此：

```
parentScore =bestChildScore*(1+0.12 * extraHits+0.10 * multiChannelHit)
```

---

体现：

### 多命中奖励

```
更多证据=更可信
```

---

### 多通道奖励

```
向量命中+关键词命中
```

可信度更高。

## Q11：这个项目最大的技术难点是什么？

推荐回答：

> 最大难点不是切块，而是如何同时保证检索精度和上下文完整性。
> 
> 如果Chunk切小，召回率高但上下文丢失；
> 
> 如果Chunk切大，上下文完整但Embedding效果下降。
> 
> 所以最终设计了父子块架构：
> 
> - 子块负责召回
> - 父块负责补全
> - parent_block_id建立关联
> - elevateToParentBlocks恢复完整语义单元
> 
> 这是整个项目最终解决的核心矛盾。
