# 第一类：为什么要做多层防护

### Q1：为什么 Agent 要做这么多层限流？只做 Hook 限流不行吗？

### 回答

单层限流无法覆盖所有风险场景，因此采用纵深防御设计。

例如：

- Hook 层解决 Agent 无限思考（LLM Loop）
- Retry 层解决工具偶发失败
- JVM Registry 解决单机重复执行
- Redis Lease 解决集群并发执行
- AOP 幂等解决用户重复提交
- HTTP Guard 解决非法请求

假设 Hook 失效：

Agent 可能继续执行。

但 Redis Lease 仍然保证：

```
同一个 conversationId只能有一个节点持有执行权
```

因此不会演变成集群雪崩。

每层解决不同维度的问题，而不是重复限流。

---

### Q2：为什么超限选择 END 而不是 THROW？

### 回答

这是用户体验和系统稳定性的权衡。

如果采用：

```
ExitBehavior.THROW
```

用户会看到：

```
Tool call limit exceeded
```

整个会话直接失败。

而：

```
ExitBehavior.END
```

会让 Agent 使用已有推理结果结束流程。

例如：

```
用户：分析某公司财报
```

Agent 已经完成 90% 推理。

此时达到 Tool Limit。

END 策略：

```
返回已有分析结果
```

THROW 策略：

```
直接报错
```

因此选择优雅终止。

---

# 第二类：循环调用问题

### Q3：Agent 为什么会出现循环调用？

### 回答

ReAct Agent 的执行模式是：

```
Thought ↓Action ↓Observation ↓Thought
```

如果模型始终认为：

```
信息不够
```

就可能无限执行：

```
search ↓search ↓search ↓search
```

形成循环。

因此需要：

```
ModelCallLimitHookToolCallLimitHook
```

从框架层强制截断。

---

### Q4：如果 Hook 失效怎么办？

### 回答

项目中还有第二层防护。

例如：

```
Hook失效 ↓Tool Retry触发 ↓HTTP Timeout触发 ↓Redis Lease过期 ↓任务被回收
```

其中 Tavily：

```
connect timeout = 3sread timeout = 6s
```

即使 Agent 不断调用工具，也无法无限占用资源。

---

# 第三类：指数退避

### Q5：为什么不用固定重试间隔？

### 回答

固定间隔容易产生重试风暴。

例如：

```
1000个请求同时失败
```

如果固定：

```
500ms
```

那么：

```
500ms后再次同时发起
```

形成惊群效应（Thundering Herd）。

因此采用：

```
200ms400ms800ms
```

指数退避。

再加：

```
jitter=true
```

实际可能变成：

```
178ms425ms763ms
```

请求会被打散。

---

### Q6：为什么最大重试次数设置为 2？

### 回答

因为 Tavily 查询属于外部依赖。

统计发现：

- 第一次失败大部分是网络抖动
- 第二次成功率明显提升
- 第三次以后收益非常低

而延迟增加明显。

因此：

```
最大重试 = 2
```

在成功率和响应时间之间取得平衡。

# 第四类：幂等设计

### Q7：为什么要 Redis Success Flag + Redisson Lock 双层设计？

### 回答

两者解决的问题不同。

---

Redisson Lock：

解决

```
正在执行
```

的问题。

例如：

```
请求A请求B同时到达
```

只有一个获得锁。

---

Success Flag：

解决

```
已经执行完成
```

的问题。

例如：

```
用户刷新页面再次提交
```

即使锁已经释放：

```
successFlag存在
```

直接返回。

因此：

```
Lock 防并发SuccessFlag 防重复
```

缺一不可。

---

### Q8：为什么本地还要加 LocalLockCache？

### 回答

减少 Redis 压力。

流程：

```
先查 JVM Cache ↓再查 Redis
```

同一节点短时间大量重复请求时：

```
直接本地拒绝
```

避免每次都访问 Redis。

属于热点优化。

---

# 第五类：Redis Lease

### Q9：为什么不用 Redisson Lock，而自己实现 Lease？

### 回答

因为 Agent 执行时间不可预测。

可能：

```
10秒30秒5分钟
```

普通锁不适合长时间持有。

项目采用：

```
30秒TTL10秒续期
```

模式。

即：

```
Lease
```

而非传统锁。

这样即使服务崩溃：

```
TTL到期自动释放
```

不会形成死锁。

---

### Q10：为什么 Renew 和 Release 要校验 Token？

### 回答

防止误释放。

场景：

```
节点A获得锁 ↓GC暂停 ↓锁过期 ↓节点B获得锁
```

此时：

A恢复执行。

如果直接：

```
DEL key
```

会把 B 的锁删掉。

因此 Lua：

```
if redis.call("get",key)==tokenthen deleteend
```

只有持有者才能操作。

---

# 第六类：Checkpoint

### Q11：Checkpoint 保存什么内容？

### 回答

保存完整 Agent 状态。

包括：

```
{  "messages": [],  "currentNode": "",  "nextNode": "",  "toolResults": []}
```

存储到：

```
GRAPH_CHECKPOINT
```

表。

这样恢复时不只是聊天记录。

而是恢复整个推理现场。

---

### Q12：Checkpoint 为什么选 MySQL 而不是 Redis？

### 回答

Checkpoint 属于持久化状态。

要求：

```
应用重启不丢机器宕机不丢
```

Redis 更适合缓存。

而 MySQL：

```
ACID持久化支持历史回溯
```

更适合作为状态存储。

---

### Q13：恢复是从头执行还是接着执行？

### 回答

接着执行。

执行时：

```
RunnableConfig.threadId(conversationId)
```

框架会读取：

```
该 threadId 最新 checkpoint
```

恢复：

```
messagesnodestate
```

然后从：

```
nextNode
```

继续运行。

而不是重新开始。

---

# 第七类：高频追问（面试官最爱）

### Q14：如果 Checkpoint 刚写 MySQL 就宕机怎么办？

回答：

MySQL 事务提交成功后才认为 Checkpoint 保存成功。

如果在提交前宕机：

```
恢复到上一个Checkpoint
```

最多损失最近一步推理状态。

不会导致状态损坏。

---

### Q15：如果 MySQL 挂了怎么办？

回答：

Agent 仍然能执行。

只是：

```
Checkpoint失效无法断点恢复
```

属于降级运行。

不会影响核心聊天功能。

---

### Q16：整个项目你认为最核心的设计是什么？

这是面试官最喜欢的总结题。

推荐回答：

> 我认为核心不是限流本身，而是纵深防御思想。Agent 的风险来源于模型的不确定性，因此不能依赖单点防护。我设计了 Hook 限流、指数退避、异常降级、JVM Registry、Redis Lease、AOP 幂等和 Checkpoint 恢复六层机制。即使某一层失效，其他层仍能兜底，最终保证 Agent 不会无限循环、不会重复执行、不会因为外部工具异常而导致整个服务失控。这也是整个风险管控体系的设计目标。
